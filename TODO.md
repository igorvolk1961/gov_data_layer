## P3 — v0.3.0 (концепция)

### Мотивация

Текущая архитектура v0.2.0 неработоспособна в продакшн:

1. **PDF → OCR — bottleneck.** Все документы в формате PDF с подписью. OCR каждого — минуты. Для тысяч — неприемлемо.
2. **Нет консолидированных версий.** Источник содержит только изменения, не итоговые версии (их делают Консультант+ и Гарант за деньги).
3. **Полный предварительный инжест неэффективен.** Хранить тексты всех документов дорого.
4. **API источника предоставляет названия в JSON** — без OCR. Ключевой ресурс для ускорения.

### Воркеры

Система переходит на асинхронную очередь обработки с **5 воркерами**:

#### 1. RSS Worker
Получает свежие документы из RSS-ленты источника → стек свежих документов.

#### 2. Scanner Worker
Просматривает в источнике ранее не прочитанные документы → стек старых документов.

#### 3. Metadata Worker (ключевая инновация)
Загружает метаданные документа из JSON API (без OCR!):
- **NER** для определения региона из названия
- **Regexp** для выявления отменяемых/изменяемых документов из названия
- **Title embedding** — векторное представление названия
- **Рубрикация** — спуск по иерархии рубрик от корня до максимальной релевантности
- **Регистрация** документа в БД и Qdrant
- Зависимые документы ищутся в БД; если не найдены — поиск в источнике и помещение в стек старых

#### 4. OCR Worker
Просматривает документы в стеках, выполняет OCR, **не сохраняя текст** (экономия хранилища). Выявляет ссылки на изменяемые/отменяемые документы → стек старых.

#### 5. Ingest Worker
Полный пайплайн (чанкинг → эмбеддинг → Qdrant + PostgreSQL) для документов из очереди инжеста.

### Поведение при запросе

1. **Поиск по чанкам** (как в v0.2.0) — если найдены → ответ
2. **Fallback: поиск по названиям** — векторная близость названий зарегистрированных документов к запросу
3. Если найден зарегистрированный, но не проинжещенный документ → **очередь инжеста**
4. **Оценка времени ожидания** → агент получает «документ будет готов через N минут»
5. **Polling** (опционально) — цикл опроса с таймаутом N × запас
6. Если не найден ни на одном шаге → честный ответ с датой последнего зарегистрированного документа

### Какие пайплайны переходят из v0.2.0

| Пайплайн v0.2.0 | Переходит | Изменения |
|-----------------|-----------|-----------|
| **Search** (Metadata Routing) | ✅ Без изменений | Qdrant + payload-фильтрация |
| **Document Detail** | ✅ Без изменений | Чанки + цитаты |
| **Ingest** | ⚠️ Частично | Chunking + embedder + Qdrant — те же. Добавляется очередь, lazy pull, polling |
| **OCR** | ⚠️ Меняется | Был обязательным → отдельный воркер без сохранения текста |
| **Metadata extraction** | ❌ Новый | NER + regexp + title embedding |
| **Dependency graph** | ❌ Новый | Граф «документ A отменяет/изменяет B» |

### Оценка влияния

- **Ядро (ODLService, модели)** — почти без изменений
- **Qdrant** — добавляется коллекция title-векторов
- **PostgreSQL** — таблицы `ingest_queue`, `document_dependency`
- **Адаптеры** — добавляется `get_metadata(document_id)` (без OCR)
- **Новое** — 5 воркеров + очередь + polling
