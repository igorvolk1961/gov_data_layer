# CHANGELOG: v0.1.0 → v0.2.0

**Дата:** 2026-07-22
**Статус:** In development (branch `develop`)

---

## Основные изменения

### 1. Metadata Routing: payload-поля заполнены

**v0.1.0:** Чанки в Qdrant содержали только `text` и `embedding`. Поля `region_id`, `topic_ids`, `organization` — пустые. Фильтрация не работала.

**v0.2.0:** При инжесте заполняются:

| Поле | Откуда берётся |
|------|---------------|
| `region_id` | Из организации через `organization.jurisdiction_id` / `region_id` |
| `topic_ids` | Через `link_chunks_to_topics()` — косинусная близость чанка к рубрикам |
| `legal_status` | `ACTIVE` для stub-документов (production — из JSON API) |
| `organization` | Из метаданных документа |

**Результат:** Фильтрация по региону, рубрике и юридическому статусу теперь работает.

---

### 2. ConfidenceSignals: разложенные сигналы

**v0.1.0:** Три поля: `retrieval_relevance`, `data_freshness`, `source_availability`.

**v0.2.0:**

| Поле | v0.1.0 | v0.2.0 |
|------|--------|--------|
| `retrieval_relevance` | `float` — raw Qdrant score | Без изменений |
| `topic_relevance` | Отсутствует | **Добавлен:** `list[TopicMatch]` — (`topic_id`, `score`) для топ-3 рубрик, ближайших к запросу |
| `source_availability` | `enum AVAILABLE/DEGRADED/UNAVAILABLE` | **Удалён.** В архитектуре с предварительным инжестом (данные в Qdrant/PG) live-доступ к источнику на момент запроса не требуется |
| `data_freshness` | `valid_from` документа | **Переопределён** как `last_verified_at: datetime \| None` — дата последней успешной верификации данных в индексе |

**Мотивация изменений:**
- **`topic_relevance`** — разложенный сигнал по принципу mechanism/policy: слой вычисляет, агент решает, какой вес придавать тематическому совпадению
- **`source_availability`** — не имеет смысла: данные уже загружены в индекс, источник на момент запроса не участвует
- **`data_freshness`** → `last_verified_at` — дата подписания документа (`valid_from`) не говорит агенту о свежести копии. Дату инжеста тоже. Полезный сигнал: «данные проверялись N времени назад»

---

### 3. LegalStatus: упрощение для гражданского сценария

**v0.1.0:** Четыре статуса: `ACTIVE`, `REVOKED`, `MODIFIED`, `UNKNOWN`.

**v0.2.0:** Два статуса:
- `ACTIVE` — документ действует (или вступит в силу в будущем)
- `NOT_ACTIVE` — документ отменён/утратил силу

Изменения:
- Удалена таблица `document_section_modification` (история изменения отдельных разделов не нужна для гражданского сценария)
- Упрощена модель: `LegalStatus.UNKNOWN` и `LegalStatus.MODIFIED` заменены на `NOT_ACTIVE`

## Известные проблемы (v0.2.0)

### P1 (блокеры для production)

| # | Проблема | Описание |
|---|----------|----------|
| 1 | **OCR-зависимость** | Все 6 документов требуют предварительного OCR. Для production нужен конвейер без OCR (через API pravo.gov.ru) |
| 2 | **Stub-режим** | PravoAdapter работает в режиме заглушки. Production-режим не протестирован end-to-end |
| 3 | **Docker image не собран** | Образ не протестирован — может не запуститься |

### P2 (качество)

| # | Проблема | Описание |
|---|----------|----------|
| 4 | **`total_count` может быть неточным** | При большом количестве чанков Qdrant может вернуть приблизительное значение |
| 5 | **Нет SLO-замеров** | Латентность, токен-бюджет, свежесть — не измерены |
| 6 | **`_check_missing_region` — заглушка** | Детекция отсутствующего региона не работает |
| 7 | **Два источника истины для схемы БД** | Liquibase XML + Python get-or-create |

### P3 (v0.3.0)

| # | Проблема | Описание |
|---|----------|----------|
| 8 | **PDF → OCR — bottleneck** | Для полноценного инжеста тысяч документов OCR неэффективен |
| 9 | **Нет консолидированных версий** | Источник не предоставляет итоговые версии документов (только изменения) |
| 10 | **Нет ленивого инжеста** | Все документы инжестятся сразу, а не по запросу |
| 11 | **Ручное выявление зависимостей** | Связи «документ A отменяет/изменяет документ B» не детектируются автоматически |

## План v0.3.0 (концепция)

Ключевые направления развития:

1. **RSS-лента** — первичное обнаружение свежих документов
2. **Title-based metadata extraction** — NER для региона, regexp для зависимостей (без OCR)
3. **Ленивый инжест** — документы инжестятся по запросу (pull model)
4. **Очередь инжеста** — с оценкой времени ожидания для агента
5. **Детекция зависимостей** — из названий документов («О внесении изменений в...» → target)
6. **Polling-механизм** — агент может дождаться результата с таймаутом
