# Анализ необходимости вынесения persistence-слоя в отдельный контейнер

## 1. Постановка задачи

Проанализировать, стоит ли выделить persistence-слой (PostgreSQL + DatabaseClient + репозитории) в отдельный микросервис/контейнер, отделив его от основного приложения (`app`).

---

## 2. Текущая архитектура

### 2.1 Компоненты persistence-слоя

```
core/persistence/
├── __init__.py                    # Экспорт DatabaseClient
├── db_client.py                   # asyncpg connection pool wrapper
├── repository/
│   ├── __init__.py                # Экспорт всех репозиториев
│   ├── document_repo.py           # CRUD документов (425 строк)
│   ├── reference_repo.py          # CRUD справочников (266 строк)
│   ├── section_repo.py            # CRUD разделов (168 строк)
│   └── change_tracking_repo.py    # Трекинг изменений (203 строки)
└── migrations/
    ├── changelog-root.xml
    └── v001/
        ├── changelog.xml
        └── 000-006_create_*.sql    # 7 SQL-файлов миграций
```

### 2.2 Текущая интеграция в docker-compose.yml

```yaml
services:
  metadata-db: # PostgreSQL — отдельный контейнер (БД)
    image: postgres:16-alpine
    container_name: odl-metadata-db

  liquibase: # Миграции — отдельный контейнер (init)
    image: liquibase/liquibase:4.28
    depends_on: [ metadata-db ]

  app: # Основное приложение — содержит persistence-логику
    build: ..
    depends_on: [ metadata-db ]
    environment:
      - DATABASE_URL=postgresql://odl:odl@metadata-db:5432/odl_metadata
```

### 2.3 Текущая интеграция в коде

В [`core/main.py`](core/main.py:134-142):
```python
db: DatabaseClient | None = None
if app_config.database_url:
    db = DatabaseClient(dsn=app_config.database_url)
```

В [`core/odl_service.py`](core/odl_service.py:58-72):
```python
class ODLService:
    def __init__(self, adapters, tracer=None, cache=None, db=None):
        self._db = db
        self._doc_repo = DocumentRepository(self._db, ref_repo) if db else None
        # ... lazy init остальных репозиториев
```

**Ключевой момент:** persistence — это **библиотека/пакет**, встроенный в `app`, а не отдельный сервис. Репозитории вызываются синхронно в том же процессе, что и ODLService.

## 4. Критерии оценки

### 4.1 Связанность (Coupling)

**Текущая ситуация:**
- Persistence-слой жёстко связан с `ODLService` через импорт: [`core/odl_service.py:30-36`](core/odl_service.py:30)
- Репозитории принимают `DatabaseClient` через DI (слабая связанность на уровне классов)
- Но вызовы репозиториев — прямые (in-process), без сетевого прослойки

**Оценка:**
- Связанность **средняя** — DI есть, но это in-process вызовы
- Выделение в отдельный сервис (Вариант B) **увеличит** связанность (сетевые вызовы, сериализация, контракты)
- Выделение в отдельный пакет (Вариант C) **не изменит** связанность

### 4.2 Масштабируемость

**Текущая ситуация:**
- `app` — монолит: REST API + MCP + persistence в одном процессе
- Единственный потребитель persistence — `ODLService`
- Нет сценария, где persistence нужен отдельно от `app`

**Оценка:**
- **Нет потребности** в независимом масштабировании persistence
- Если persistence станет узким местом, проще увеличить pool соединений asyncpg (`_POOL_MAX_SIZE: int = 10` в [`core/persistence/db_client.py:26`](core/persistence/db_client.py:26))
- Выделение persistence в отдельный сервис **добавит latency** (сетевой hop) без выигрыша в масштабировании

### 4.3 Изолированность сбоев (Failure Isolation)

**Текущая ситуация:**
- Если `metadata-db` недоступен, `DatabaseClient.connect()` падает с `ConnectionError`
- В [`core/odl_service.py:130-131`](core/odl_service.py:130): `if self._db is None: return` — graceful degradation
- В [`core/main.py:136-140`](core/main.py:136): `if app_config.database_url:` — DB опциональна

**Оценка:**
- **Уже есть graceful degradation** — DB опциональна, при недоступности persistence пропускается
- Выделение persistence в отдельный сервис **добавит новые точки отказа**: сеть, DNS, HTTP/gRPC таймауты
- In-process вызов asyncpg **надёжнее** сетевого вызова к persistence-сервису

### 4.4 Производительность (Latency)

**Текущая ситуация:**
- In-process: `ODLService → DocumentRepository → DatabaseClient → asyncpg → PostgreSQL`
- Один сетевой hop (app → metadata-db)
- Нет накладных расходов на сериализацию/десериализацию между сервисами

**Вариант B (отдельный сервис):**
- `ODLService → HTTP/gRPC → Persistence API → DatabaseClient → asyncpg → PostgreSQL`
- **Два** сетевых hop вместо одного
- Накладные расходы на HTTP/gRPC сериализацию (JSON/Protobuf)
- Для каждого `upsert_document()` (16 полей + M:N) — дополнительная сериализация

**Оценка:**
- **Значительное ухудшение** производительности без явной выгоды
- Для фонового ингеста документов — некритично, но для API-запросов — заметно

### 4.5 Сложность разработки и поддержки

**Текущая ситуация:**
- 4 файла репозиториев, ~1062 строки кода
- 1 файл DatabaseClient, ~162 строки
- Всё в одном Python-пакете `core.persistence`

**Вариант B (отдельный сервис):**
- Нужен новый сервис с API-контрактом (OpenAPI/Protobuf)
- Нужна аутентификация между сервисами
- Нужна обработка ошибок сети (retry, timeout, circuit breaker)
- Нужна версионирование API
- Нужен отдельный CI/CD, Dockerfile, healthcheck
- Удвоение инфраструктуры (логи, метрики, трассировка)

**Оценка:**
- **Значительное усложнение** без пропорциональной выгоды
- Для 10 таблиц и 4 репозиториев — overhead неоправдан

### 4.6 Переиспользуемость

**Текущая ситуация:**
- Persistence используется **только** `ODLService`
- Нет других потребителей (других микросервисов), которым нужен доступ к PostgreSQL

**Оценка:**
- **Нет потребности** в переиспользовании persistence как сервиса
- Если в будущем появится второй потребитель (например, отдельный ingest-worker), он может:
  - Импортировать `core.persistence` как библиотеку (Вариант C)
  - Использовать общий `DatabaseClient` с тем же connection pool

### 4.7 Безопасность

**Текущая ситуация:**
- `DATABASE_URL` передаётся через environment variable в [`docker-compose.yml:125`](docker-compose.yml:125)
- Пароль `odl` — не секрет (dev-окружение)

**Вариант B (отдельный сервис):**
- Нужна аутентификация между `app` и `persistence` (mTLS, API key, JWT)
- Нужна авторизация (какие операции разрешены)
- Нужна защита от SSRF

**Оценка:**
- **Усложнение безопасности** без явной необходимости
- В текущей архитектуре БД изолирована в своей сети Docker (не暴露 порты наружу)

## 6. Анализ сценариев "что, если"

### 6.1 Если появится второй потребитель PostgreSQL

**Сценарий:** Отдельный ingest-worker, который загружает документы в фоне.

**Решение без отдельного persistence-сервиса:**
- Ingest-worker импортирует `core.persistence` как библиотеку
- Использует тот же `DatabaseClient` с отдельным connection pool
- Никаких изменений в API не требуется

**Почему это проще:**
- Ingest-worker уже существует как логический компонент (см. [`plans/container.md:19`](plans/container.md:19))
- Он может быть выделен в отдельный контейнер, но persistence остаётся библиотекой
- Общий доступ к БД через asyncpg pool — стандартная практика

### 6.2 Если persistence-логика станет очень сложной

**Сценарий:** 30+ таблиц, сложные бизнес-правила на уровне persistence.

**Решение:**
- Выделить `core.persistence` в отдельный Python package (`gov-persistence`)
- Публиковать в приватный PyPI
- Версионировать независимо от `gov-data-layer`

**Почему не отдельный сервис:**
- Сложность бизнес-логики не требует сетевой границы
- Транзакционная целостность проще в in-process (один connection pool)
- Сетевой API для сложных запросов (recursive CTE, иерархические выборки) — это боль

### 6.3 Если нужна строгая изоляция данных

**Сценарий:** Мультитенантность, разные уровни доступа к данным.

**Решение:**
- Отдельный persistence-сервис с авторизацией на уровне API
- Row-level security в PostgreSQL
- Но это **гипотетический** сценарий для текущего проекта

## 7. Рекомендация

### Вердикт: **НЕ выносить persistence в отдельный контейнер**

Persistence-слой должен остаться **in-process библиотекой** в составе `app` контейнера (Вариант A).

### Обоснование

1. **Нет потребителей вне ODLService** — persistence используется только одним клиентом. Сетевой API не даёт выигрыша.

2. **Производительность** — in-process asyncpg вызовы быстрее и надёжнее, чем HTTP/gRPC к отдельному сервису. Для каждого `get_document_detail()` сейчас выполняется ~5 SQL-запросов (документ + организации + темы + секции). С отдельным сервисом каждый запрос — это 2 сетевых hop вместо 1.

3. **Graceful degradation уже реализован** — [`ODLService`](core/odl_service.py:130-131) корректно работает без БД. Выделение persistence в сервис **ухудшит** отказоустойчивость (добавит новые точки отказа).

4. **Сложность неоправданна** — 10 таблиц, 4 репозитория, ~1200 строк кода. Overhead отдельного сервиса (API-контракт, аутентификация, CI/CD, логи, метрики) превышает размер самого persistence-слоя.

5. **Транзакционная целостность** — in-process проще обеспечить атомарность операций (документ + M:N связи + секции). С отдельным сервисом нужен distributed transaction или saga pattern.

6. **Container diagram уже отражает** текущую архитектуру — persistence как часть `app`, а не отдельный контейнер (см. [`plans/container.md`](plans/container.md)).

### Что можно улучшить без выделения в отдельный сервис

| Улучшение | Описание | Приоритет |
|---|---|---|
| **Выделить `core.persistence` в отдельный Python package** | Отдельный репозиторий, версионирование, CI | Низкий (когда появится второй потребитель) |
| **Добавить `transaction()` context manager** | В [`DatabaseClient`](core/persistence/db_client.py) для атомарных операций | Средний (уже запланирован в [`plans/persistence_`](plans/persistence_)) |
| **Generic `upsert()` helper** | Уменьшить boilerplate в репозиториях | Средний (уже запланирован) |
| **Улучшить мониторинг** | Метрики времени выполнения запросов, pool utilization | Низкий |

## 8. Заключение

**Вынесение persistence-слоя в отдельный контейнер — преждевременная оптимизация**, которая:
- Усложнит архитектуру без явной выгоды
- Ухудшит производительность (дополнительный сетевой hop)
- Увеличит поверхность для сбоев (сеть, DNS, HTTP)
- Потребует значительных трудозатрат на API-контракт, аутентификацию, CI/CD

Текущая архитектура (in-process persistence как библиотека) — **правильный выбор** для текущего этапа проекта. Если в будущем появится реальная потребность (второй потребитель, мультитенантность), persistence можно выделить в отдельный пакет без изменения архитектуры взаимодействия.
