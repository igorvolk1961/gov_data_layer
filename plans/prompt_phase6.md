# Промпт для Kilo Code (режим Архитектор)
## Роль и контекст

Ты — архитектор Python-проекта, помогающий довести до конца тестовое задание на позицию Senior/Lead Engineer. Задание: **переносимый слой официальных данных для AI-агентов** (сервис, к которому LLM-агент обращается вместо веб-поиска по вопросам госуслуг, пособий, нормативки). Слой отдаёт ответ с provenance (цитата, источник, дата, юридический статус, разложенные сигналы уверенности), а если официального основания нет — честно сообщает.

**Срок:** середина недельного срока, осталось ~3.5 дня.
**Оценка:** не «запустилось/не запустилось», а зрелость решения. Критерии:
- Архитектура и переносимость (Gen-AI ready API, MCP/OpenAPI, механизм/политика) — **~35%**
- Работающая вертикаль end-to-end + шов адаптера — **~20%**
- Инженерная культура (репозиторий, CI, тесты, секреты, наблюдаемость, типизированные ошибки, graceful degradation) — **~25%**
- Надёжность и SLO (холодный/горячий старт, кэш, поведение при сбое) — **~10%**
- Достоверность и ограничения (цитирование, юр.статус vs свежесть, отказ) — **~10%**

**Принцип:** лучше одна вертикаль, доведённая до чистого состояния, чем три наполовину. Широта — в спецификации, глубина — в коде.

## Принятые решения для следующей фазы

### 1. Рабочая вертикаль: только Минтруд
Ограничиваюсь документами Министерства труда и социальной защиты:
- URL блока: `http://publication.pravo.gov.ru/documents/block/foiv281`
- Это снижает неопределённость: один тип документов, одна юрисдикция, предсказуемая структура

### 2. OCR через Yandex Cloud Vision
- PDF на pravo.gov.ru — скан-копии, текстового слоя нет
- **Решение:** использовать Yandex Cloud Vision OCR как основной провайдер извлечения текста
(Идентификатор ключа =<your-ya-key-id>
секретный ключ =<your-ya-key-secret>
folder_id =<your-ya-folder-id>)
- **Важно:** провайдер OCR должен быть **настраиваемым через config** (интерфейс `OCRProvider` с реализациями `YandexVisionOCR`, `TesseractOCR` (CPU), `StubOCR` для тестов). Это соответствует принципу модель-агностичности из ТЗ.

### 3. Стратегия снижения риска через заглушку
Полный пайплайн с OCR — это риск (внешний сервис, стоимость, латентность, качество). Поэтому:

**Режим демонстрации (stub mode):**
- `PravoAdapter` в режиме заглушки:
метод начальной загрузки `initial_ingest()` загружает 3 документа:
1. Приказ Министерства труда и социальной защиты Российской Федерации от 29.09.2020 № 668н
http://publication.pravo.gov.ru/document/0001202012230060
2. Приказ Министерства труда и социальной защиты Российской Федерации от 21.03.2022 № 154н
http://publication.pravo.gov.ru/document/0001202206200030
3. Постановление Правительства РФ от 16 декабря 2022 года № 2330
http://publication.pravo.gov.ru/document/0001202212190143

метод `ingest_new()` возвращает **фиксированный список URL** на 2 документа Минтруда:
1. Приказ Министерства труда и социальной защиты Российской Федерации от 03.06.2026 № 238н
http://publication.pravo.gov.ru/document/0001202607060006
2. Приказ Министерства труда и социальной защиты Российской Федерации от 08.05.2026 № 200н
http://publication.pravo.gov.ru/document/0001202606090026

- Эти документы проходят полный пайплайн: скачивание PDF → OCR → нормализация → эмбеддинг → запись в Qdrant + метаданные в Postgresql
- После этого я могу задавать слою запросы, ответы на которые **гарантированно есть** в векторной базе
- Это позволяет показать работающую вертикаль end-to-end

**Режим продакшена (production mode):**
- Переключается через config (`INGEST_MODE=stub|production`)
- В production: реальный мониторинг всех исочников, заданных в конфигурации (пока толко RSS/API pravo.gov.ru) + OCR всех новых документов
- В stub: фиксированный список + предзаполненная база

Описание API на странрце http://publication.pravo.gov.ru/help
Для источников, поддерживающих rss - отдельый класс RSSAdapter. PravoAdapter наследует от него.
(часть функционала RSS можно было бы реализовать и только спецуифичесими средствами API pravo.gov.ru. Но мне важно найти наиболее общее решение - отрази это в ADM.md)
Предусмотри отдельный класс для начальной загрузки и REST-endpoint для ее выполнения
Также предусмотри REST-endpoints для получения информации о состоянии векторной базы, и о перечне источников (адаптеров)
Необходимо также доработать каноническую модель OfficialDocument в соответствии с описанием API на странице
http://publication.pravo.gov.ru/help (эта модель частная, но представительная, поэтому ее можно использовать как каноническую  - отрази это в файле ADM.md)


### 4. Пайплайн ingest (полный)
```
RSS/API pravo.gov.ru (или фиксированный список в stub)
    ↓
Скачивание PDF
    ↓
OCRProvider.extract_text(pdf) → текст
    ↓
Нормализация в OfficialDocument (каноническая модель)
    ↓
Чанкинг (DocStructSplitter - https://github.com/igorvolk1961/smart_chunker)
    ↓
Эмбеддинг (bge-m3 под ollama)
    ↓
Запись: Qdrant (вектор + payload) + Postgresql (метаданные + рубрикатор)
```
Обоснование выбора DocStructSplitter - собственная разработка, ориентированная на тексты с нумерованными разделами, характерные для официальных документов.
Обоснование выбора Postgresql: Есть опыт использования, удобен для обработки иерархических связей рубрик (СТЕ)
Обоснование выбора bge-m3 - высокая точость на русском языке и большой контекст. Минус- медленная, но для тестового задания обработка больших оъемов  не требуется. В продакшн скорость поступления новых документов в источниках ожидается низкой, а именно обработка новых документов требует многократных обращений к ембеддеру. Пользовательские запросы вряд ли потребуют больше одного обращения к эмбеддеру на запрос. При высоких нагрузках можо будет рассмотреть переход на multilingual-e5 или nextfire/paraphrase-multilingual-minilm:l12-v2 (очень быстрый эмбеддер, но с пониженным качеством) - обоснование выбора запиши в AML.md

## Ограничения

- **Python 3.12+**
- **Нет GPU** — только CPU для локальных задач (эмбеддинги, Tesseract как fallback)
- **Yandex Cloud** —
- **Время** — ~3.5 дня, нужно фокусироваться на глубине одной вертикали
- **Токен-бюджет** — слой не генерирует свободный текст, финальную формулировку делает вызывающий агент
- **Модель-агностичность** — если внутри используется LLM, она сменяемая
