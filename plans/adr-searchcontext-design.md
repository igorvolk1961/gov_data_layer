# ADR: SearchContext — структурированный входной контракт

**Статус:** принято
**Дата:** 2026-07-10
**Контекст:** анализ класса `SearchContext` на соответствие тестовому заданию

## Проблема

Класс `SearchContext` (core/models/models.py) — структурированный входной контракт для
фильтрации и роутинга запросов к слою официальных данных. Тестовое задание
(postanovka_gov_data_layer.md) требует:

1. **Структурированный входной контракт** — текст вопроса + опциональный контекст
   (тема/интент, регион, рубрика, организация); при неполноте — мягкая деградация.
2. **Параметры наружу** — регион, рубрика, «только официальное», окно актуальности.

В ходе анализа выявлены следующие проблемы исходного дизайна:

### Проблема 1: query vs intent

`query` (свободный текст вопроса) передаётся отдельным параметром в инструменты MCP.
Выделять `intent` как отдельное поле в `SearchContext` — избыточно и вредно:
агент будет путаться, куда писать намерение пользователя.

**Решение:** `query` — это и есть интент. Не дублировать.

### Проблема 2: пересечение region / jurisdiction / organization

Поля `region` (география), `jurisdiction` (уровень власти) и `organization` (орган)
семантически пересекаются. Пример: «законы законодательного собрания Московской области»:

- `region` = «Московская область»
- `organization` = «Законодательное собрание Московской области»
- `jurisdiction` = «региональная»

Агент не является экспертом-правоведом. Чем больше семантически пересекающихся полей,
тем выше вероятность ошибок заполнения и лишних токенов на размышления.

**Решение:** убрать `jurisdiction` из `SearchContext`. Юрисдикция — производная
характеристика, которую слой вычисляет сам из `region` и `organization`.
Это прямое применение принципа механизм/политика.

### Проблема 3: отсутствие обязательных по заданию полей

В задании явно перечислены параметры, которых нет в `SearchContext`:
- `official_only` — «только официальное»
- `max_age_days` — окно актуальности
- `page` — пагинация (токен-осознанность)

**Решение:** добавить все три поля.

## Решение

### Итоговая структура SearchContext

```python
class SearchContext(BaseModel):
    """Контекст запроса — опциональные параметры для роутинга и фильтрации.

    query передаётся отдельным параметром в инструменты MCP и содержит
    свободный текст вопроса/интент пользователя. SearchContext содержит
    только структурированные параметры для фильтрации и роутинга.
    """

    region: str | None = Field(
        default=None,
        description="Географический регион (город, область, край). Пример: 'Московская область', 'г. Москва'",
    )
    topic: str | None = Field(
        default=None,
        description="Тематическая рубрика. Пример: 'налоги', 'социальное обеспечение', 'земельное право'",
    )
    organization: str | None = Field(
        default=None,
        description="Орган, принявший документ. Пример: 'ФНС', 'Законодательное собрание Московской области'",
    )
    official_only: bool = Field(
        default=False,
        description="Только официальные источники (без аналитики и комментариев)",
    )
    max_age_days: int | None = Field(
        default=None,
        ge=1,
        description="Максимальный возраст документа в днях. Пример: 30 — только документы не старше 30 дней",
    )
    max_results: int = Field(
        default=10,
        ge=1,
        le=50,
        description="Максимальное количество результатов (пагинация)",
    )
    page: int = Field(
        default=1,
        ge=1,
        description="Номер страницы результатов",
    )
```

### Что изменилось относительно текущей реализации

| Поле | Было | Стало | Причина |
|------|------|-------|---------|
| `region` | ✅ | ✅ | Без изменений |
| `topic` | ✅ | ✅ | Без изменений |
| `organization` | ✅ | ✅ | Без изменений |
| `max_results` | ✅ | ✅ | Без изменений |
| `jurisdiction` | ❌ | — | **Удалено.** Вычисляется слоем из region/organization |
| `official_only` | ❌ | ✅ **Добавлено** | Требование задания |
| `max_age_days` | ❌ | ✅ **Добавлено** | Требование задания |
| `page` | ❌ | ✅ **Добавлено** | Пагинация для токен-осознанности |

### Что НЕ вошло и почему

- **`intent`** — не дублируется, т.к. `query` уже является интентом.
- **`filters: dict[str, str]`** — отклонено как анти-паттерн: потеря строгой валидации
  Pydantic, неопределённая семантика для агента.
- **`jurisdiction`** — удалено как производная характеристика, вычисляемая слоем.

## Последствия

1. **Model-friendly API** — 7 простых полей с понятными названиями, без семантических
   пересечений. Агент заполняет то, что понимает, остальное слой выводит сам.
2. **Строгая валидация** — Pydantic проверяет типы и границы всех полей.
3. **Мягкая деградация** — все поля опциональны, слой работает best-effort.
4. **Токен-осознанность** — пагинация через `max_results` + `page`.
5. **Требуется доработка роутера** — при реализации роутинга (Phase 5) слой должен
   вычислять `jurisdiction` из `region` и `organization`.
